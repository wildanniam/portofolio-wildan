# V5 Phase 1 - Global Audit and Acceptance Gate

## Purpose

Phase 1 is the design contract for Portfolio V5. It exists so the next implementation phases do not drift into random visual polishing, isolated hero tweaks, or generic portfolio sections.

V5 succeeds only when the full portfolio experience implements the decision-maker brief, not when the hero merely has a nicer 3D object.

## Source Brief

- **Positioning:** Wildan Syukri Niam, AI Researcher & Web3 Builder.
- **Feeling:** Standing inside a living autonomous system.
- **Audience:** AI researchers or AI/Web3 founders skeptical of student portfolios.
- **Anti-audience:** People looking for a polite student CV, or people who do not value innovation, research, product planning, and fast shipping.
- **Hero object:** Autonomous reactor core.
- **Job:** Convince.
- **Three-second memory:** There was a living reactor core with projects orbiting it, and the whole page felt like proof that this student builds real autonomous systems.

## V5 Success Definition

V5 is considered successful only if these conditions are true together:

- The page does not feel like a student CV with decorative 3D.
- The page feels like entering a serious autonomous system built by a young builder with real proof.
- The content is sharper than a normal portfolio: it explains thesis, system thinking, product ownership, and shipped evidence.
- The visual language stays coherent from hero to footer: near-black technical space, cyan/mint system signal, amber proof energy, project color only when evidence is activated.
- The layout does not collapse into equal-weight cards everywhere.
- The page uses motion and interaction to reveal evidence, not just to decorate the interface.
- The site remains lightweight: desktop may use progressive WebGL, mobile and weak devices use a premium fallback.

## Global Narrative

The portfolio should read as one continuous argument:

1. **Autonomous system exists.**
   The hero creates the three-second memory: a living reactor core with projects orbiting it.

2. **There is a research thesis behind it.**
   Research vectors explain the recurring technical themes: agents, trust, on-chain intelligence, and agentic payments.

3. **The thesis has shipped evidence.**
   Work is presented as mission dossiers, not equal project cards.

4. **External proof validates the work.**
   Grants, hackathon wins, and institutional recognition become a proof ledger.

5. **The builder has an operating system.**
   Product architecture, fullstack execution, AI integration, Web3 protocols, and deployment are shown as a build loop.

6. **The visitor knows what to do next.**
   Contact is direct, serious, and collaboration-oriented.

## Cut and Rewrite Rules

Cut or avoid:

- Generic skill grids that look like resume filler.
- Equal-weight project cards where every project has the same importance.
- Long chronology that feels like a CV timeline.
- Decorative dashboards that do not reveal evidence.
- Noisy chips or badges without hierarchy.
- Generic "about me" language.
- Visual panels that repeat the same rounded card shape too often.
- 3D that does not carry narrative interaction.

Rewrite toward:

- Thesis-first language.
- Evidence-first project descriptions.
- Product ownership and architecture language.
- Short labels that feel like instrumentation.
- High-signal achievements with context.
- Proof that would change a skeptical founder or researcher's mind.

## Current Section Audit

### Navigation

- **Current status:** Functional and visually consistent, but still reads like a conventional portfolio nav.
- **Decision:** Keep structure, refine later.
- **Target:** Navigation should feel like a quiet system frame, not a loud app header.
- **Phase:** Defer until global visual pass.

### Hero

- **Current status:** Left content is strong enough to keep. Right visual has the correct direction but does not yet fully deliver the decision-maker brief.
- **Decision:** Keep copy direction, redesign/polish right-side reactor execution.
- **Target:** The hero must feel like a heavy autonomous reactor object, not a Three.js demo beside text.
- **Required changes:**
  - Core must feel physical, layered, and expensive.
  - Project modules must feel like instrument modules, not colored toy blocks.
  - Idle state must be restrained: project colors should not dominate until activation.
  - Hover/click must feel like lock-on evidence, not a normal tooltip.
  - Reveal should feel slow, weighted, and cinematic.
- **Phase:** Phase 2 and Phase 4.

### Metrics Rail

- **Current status:** Useful, but it risks feeling like a common stats strip.
- **Decision:** Keep content, redesign presentation later.
- **Target:** Metrics should feel like live telemetry from the system, not marketing counters.
- **Required changes:**
  - Replace generic "telemetry" repetition with specific signal labels.
  - Give proof hierarchy: grant, wins, shipped builds, leadership.
  - Make it visually connected to hero/reactor language.
- **Phase:** Phase 5 or global section polish.

### Research Deck

- **Current status:** Content direction is aligned, but the visual treatment still uses familiar cards and a decorative central panel.
- **Decision:** Keep research vectors, redesign visual grammar.
- **Target:** Research vectors should feel like command channels connected to shipped systems.
- **Required changes:**
  - Reduce generic card feeling.
  - Make the central diagram behave like a system map, not a decorative infographic.
  - Show how each vector maps to projects and evidence.
  - Copy should become slightly sharper and more thesis-like.
- **Phase:** Phase 6.

### Work / Mission Control

- **Current status:** The "mission dossiers" concept is good. The implementation still relies on a large deck panel and stacked project buttons, so it can feel like a themed portfolio section.
- **Decision:** Keep concept, redesign interaction and hierarchy.
- **Target:** Work should be the main evidence system. A visitor should inspect projects like proof dossiers.
- **Required changes:**
  - Establish project priority: Fradium, PayGate, AgentPay, Nova, SpecHeal should not all feel identical.
  - Replace purely decorative project visual with evidence-specific artifacts.
  - Make roles, achievements, links, and architecture signals more scannable.
  - Avoid excessive stack rows as the dominant right-side content.
- **Phase:** Phase 7.

### Proof Ledger

- **Current status:** Strong concept and useful content. It still looks like achievement cards.
- **Decision:** Keep, sharpen.
- **Target:** Proof should feel like external validation ledger, not award collection.
- **Required changes:**
  - Group proof by weight: grants, wins, institutional recognition.
  - Add clearer context for why each proof matters.
  - Reduce decorative repetition.
  - Keep source notes available, but do not let them feel administrative.
- **Phase:** Phase 8.

### Builder System

- **Current status:** This is closest to a generic skill grid, which the brief explicitly wants to cut.
- **Decision:** Rewrite and redesign.
- **Target:** Replace "skills" with operating loop: research, architecture, build, validate, deploy, explain.
- **Required changes:**
  - Convert capability cards into process stages or system loop.
  - Tie each capability to proof from projects.
  - Remove list-like skill dumping.
- **Phase:** Phase 9.

### Build Log

- **Current status:** Good intent, but it can become a timeline if left as isolated entries.
- **Decision:** Keep only if it supports current momentum.
- **Target:** Build log should show active trajectory, not chronology.
- **Required changes:**
  - Keep short.
  - Make each item answer "why does this increase conviction?"
  - Consider moving into proof or system loop if it feels redundant.
- **Phase:** Phase 9.

### About

- **Current status:** Copy is aligned, but visual container is another deck panel.
- **Decision:** Keep copy direction, rewrite tighter later.
- **Target:** About should not be biography. It should state the builder operating thesis.
- **Required changes:**
  - Make it more direct and less explanatory.
  - Preserve leadership and architecture ownership.
  - Avoid sounding like a student profile.
- **Phase:** Phase 9.

### Contact

- **Current status:** Clear and useful.
- **Decision:** Keep, refine final tone.
- **Target:** Contact should invite serious collaboration, not generic networking.
- **Required changes:**
  - Make CTA feel like the next mission, not a closing footer.
  - Keep email and socials simple.
- **Phase:** Final polish.

## Content Acceptance Checklist

Before V5 is accepted, the copy must pass these checks:

- The headline and hero paragraph still sell AI Researcher & Web3 Builder clearly.
- Every major section answers one of these: thesis, evidence, proof, operating system, next action.
- No section exists only because portfolios usually have it.
- Project descriptions emphasize system thinking, role, architecture, shipping, and validation.
- Achievements explain why they matter.
- Skills are not listed as generic technologies without context.
- The copy does not sound like a polite student CV.
- The writing is professional, ambitious, and credible without sounding inflated.

## Visual Acceptance Checklist

Before V5 is accepted, the visual system must pass these checks:

- The page feels like one coherent machine-world, not isolated themed sections.
- Near-black remains dominant.
- Cyan/mint represents system signal.
- Amber represents proof/energy.
- Project colors are restrained and mostly appear during active evidence states.
- Repeated rounded cards are reduced or given stronger purpose.
- Section transitions feel atmospheric and intentional.
- The hero reactor is the most memorable object on the page.
- Motion feels slow, damped, and weighted.
- Nothing feels like generic AI dashboard decoration.

## Interaction Acceptance Checklist

Before V5 is accepted, interaction must pass these checks:

- Hero hover previews evidence smoothly.
- Click pins project evidence clearly.
- Clearing state fades out instead of snapping.
- The reactor responds as one system when a project is active.
- Work section selection feels like opening a dossier, not switching cards.
- Hover states feel responsive like Stripe, but visually restrained.
- Interactions do not make the site heavy or distracting.

## Performance Acceptance Checklist

Before V5 is accepted, implementation must pass these checks:

- Desktop WebGL loads progressively.
- Mobile does not mount WebGL.
- Reduced-motion users receive fallback visuals.
- No horizontal overflow.
- `npm run lint` passes.
- `npm run build` passes.
- Browser QA screenshots are saved for each major visual phase.
- Initial page experience is not blocked by Three.js.

## Phase Breakdown After This Gate

### Phase 2 - Hero Core Artifact Polish

Goal: Make the autonomous reactor core feel physical, expensive, and iconic.

Scope:

- Core material and geometry.
- Shell depth and light behavior.
- Idle object presence.
- No changes to global content unless spacing requires it.

Manual gate:

- Does the core finally feel like a designed artifact, not a low-poly ball?

### Phase 3 - Hero Orbital Modules Polish

Goal: Make project satellites feel like serious instrument modules orbiting the reactor.

Scope:

- Module geometry.
- Idle/active color restraint.
- Orbital rail language.
- Project identity only on activation.

Manual gate:

- Do the satellites feel like parts of an autonomous system, not toy blocks?

### Phase 4 - Hero Interaction Polish

Goal: Make hover, click, pin, evidence, and clearing feel smooth and meaningful.

Scope:

- Preview state.
- Pinned state.
- Evidence annotation.
- Camera/core response.
- Hitbox and tooltip timing.

Manual gate:

- Does interacting with a project feel like the system locking onto evidence?

### Phase 5 - Hero Cinematic Reveal and Performance QA

Goal: Make the hero entrance feel slow, weighted, and lightweight.

Scope:

- First load reveal.
- Fallback to WebGL transition.
- Mobile fallback.
- Performance checks.

Manual gate:

- Does the first viewport feel premium before any scroll?

### Phase 6 - Research System Redesign

Goal: Turn research vectors into connected command channels.

Scope:

- Research section structure.
- Copy sharpening.
- Visual connection to shipped projects.

Manual gate:

- Does this section prove there is a thesis behind the builds?

### Phase 7 - Work Evidence System

Goal: Turn project presentation into serious proof dossiers.

Scope:

- Project hierarchy.
- Evidence artifacts.
- Role and architecture clarity.
- Links and stack treatment.

Manual gate:

- Would a skeptical founder/researcher believe these were real shipped systems?

### Phase 8 - Proof Ledger Polish

Goal: Make external validation feel impossible to miss.

Scope:

- Achievement hierarchy.
- Grant/win context.
- Source proof treatment.

Manual gate:

- Does the proof feel credible, not boastful?

### Phase 9 - Builder Operating System and Final Sections

Goal: Replace generic capability sections with a serious builder operating loop.

Scope:

- Builder system.
- Build log.
- About.
- Contact.

Manual gate:

- Does the bottom half strengthen the same argument instead of becoming a normal portfolio?

## Phase 1 Manual Review Gate

Before implementation continues, the user should approve or edit these decisions:

- V5 success definition.
- Section-by-section audit decisions.
- Content cut/rewrite rules.
- Visual acceptance checklist.
- Phase order.
- Manual gate question for each phase.

Only after Phase 1 is approved should Phase 2 implementation begin.
